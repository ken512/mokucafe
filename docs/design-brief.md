# もくカフェ デザインブリーフ

Figma MCP でデザインを生成・修正する際に、このドキュメントをプロンプトに含めること。

---

## サービス概要

**もくカフェ**とは、カフェで作業仲間を見つけるアプリ。
「今日この店で一緒に作業しませんか？」という募集を投稿し、参加申請で繋がる。

### ターゲット
- カフェで作業しているが、一人だと集中できない・寂しい人
- 作業仲間を気軽に見つけたいフリーランス・学生

### ブランドの雰囲気
- **温かい・ゆるい・リアル**（ガチなビジネスマッチングではない）
- カフェの空気感：落ち着き、少しざわめき、コーヒーの香り
- 押しつけがましくない。あくまで「今日だけ一緒にどうですか？」

---

## デザイン原則

### 1. 1画面1目的
- 1つの画面に置いていいのは1つの目的のみ
- 複数ステップの操作は画面を分ける（例：入力→確認→完了）
- ユーザーが「次に何をすればいいか」迷わない状態を保つ

### 2. 導線に詰まりを作らない
- 操作完了後に必ず次の導線を用意する（ボタン・リンク）
- 「完了しました」で終わりにしない。次のアクションを明示する
- エラー時も「どうすれば解決できるか」をセットで見せる

### 3. コントラストを確保する
- テキストと背景の色差を十分につける（WCAG AA 基準以上）
- 薄いグレー文字・淡い背景の組み合わせを避ける
- プライマリボタンは背景色と明確に区別できる色にする

### 4. ボタンラベルは動詞＋目的語
| NG（汎用的） | OK（具体的） |
|------------|------------|
| 送信 | 募集を投稿する |
| 確認 | 内容を確認する |
| OK | 申請を送る |
| 登録 | アカウントを作成する |

### 5. もくカフェらしさを維持する
- AIが出す汎用UIパターンをそのまま使わない
- カフェ・作業・つながりの文脈をデザインに反映する
- アイコン・イラスト・コピーに「カフェ感」を持たせる

## 画面一覧とMVPスコープ

MVPスコープ：**募集投稿 〜 参加申請**のみ。マッチング機能は対象外。

| 画面 | パス | 目的（1つ） | 認証 |
|------|------|-----------|------|
| ホーム | / | 募集一覧を見る | 不要 |
| ログイン | /login | ログインする | 不要 |
| 新規登録 | /register | アカウントを作る | 不要 |
| 募集詳細 | /posts/[id] | 募集内容を確認して申請する | 要（申請時） |
| 募集作成 | /posts/new | 募集を投稿する | 要 |
| プロフィール | /profile | 自分の情報を編集する | 要 |

### 画面遷移（MVP）

```
ホーム（募集一覧）
  └─ 募集詳細 → 申請完了 → ホームへ戻る
  └─ 募集を投稿する → 投稿完了 → ホームへ戻る（要ログイン）

未ログイン状態でアクション → ログイン画面 → 元の画面へリダイレクト
```

## Figma MCP プロンプトテンプレート

### 新規画面を作るとき

```
以下のデザインブリーフに基づいて、[画面名] 画面を作成してください。

【サービス】もくカフェ：カフェで作業仲間を見つけるアプリ
【ブランド】温かい・ゆるい・リアル。カフェの落ち着いた雰囲気。
【この画面の目的】（1つだけ書く）
【主要コンポーネント】（必要な要素を列挙）
【完了後の導線】（次にどこへ行くか）

デザイン原則：
- 1画面1目的
- ボタンラベルは「動詞＋目的語」（例：「募集を投稿する」）
- テキストと背景は十分なコントラスト
- 完了後の導線を必ず用意する
```

### 既存画面を修正するとき

```
以下の観点でこの画面を改善してください：
1. 導線の詰まり：完了後にどこへ行けるか確認
2. ボタンラベル：「送信」「確認」を具体的な言葉に変更
3. コントラスト：薄い文字・淡い背景の組み合わせを修正
4. もくカフェらしさ：汎用UIパターンをカフェ文脈に置き換える
```
